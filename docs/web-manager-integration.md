# Web/Text Football Manager Integration Guide

This simulator works best as a match service inside your game. Your game owns persistent data such as users, clubs, players, contracts, fixtures, saves, morale, form, and finances. The simulator receives a match-day projection of that data, runs the fixture, and returns events and reports that your UI can render.

## Recommended Match Flow

1. Load the fixture from your game database.
2. Select the starting eleven and substitutes for both teams.
3. Convert your saved player attributes into simulator `Player` instances.
4. Convert each manager's tactical setup into a `Tactics` object.
5. Run `RealTimeEngine`.
6. Convert the event log into lightweight plain objects for storage.
7. Store the final score, match stats, event feed, and report.
8. Apply post-match effects in your own game systems, such as player form, fatigue, injuries, morale, league table updates, and finances.

## Data Boundary

Keep the simulator boundary small. Do not try to make simulator classes your database models.

Good persistent game data:

- `playerId`, name, shirt number, position
- attributes on a 1 to 20 scale
- current club, squad status, fitness, morale, form
- selected formation, mentality, style, press, tempo, width
- fixture id, seed, home team id, away team id

Good simulator input:

- `Player`
- `Team`
- `Tactics`
- a seeded `random` function
- optional referee profile

Good simulator output to store:

- final score
- compact event log
- headline and report sections
- selected stats
- match seed
- optionally sampled snapshots for visual replay

Avoid storing raw class instances directly. `RealTimeMatchEvent` contains `Team` and `Player` object references that are convenient in memory but usually too heavy for a database row.

## Running a Match

```ts
import RealTimeEngine from '$simulator/RealTimeEngine';
import RealTimeReporter from '$simulator/RealTimeReporter';
import Team from '$simulator/Team';

const homeTeam = new Team(true, 'Northbridge FC', homePlayers);
const awayTeam = new Team(false, 'Eastport United', awayPlayers);

const engine = new RealTimeEngine(homeTeam, awayTeam, {
    random: seededRandom(982451653),
    homeTactics: {
        formation: '4-4-2',
        style: 'high_press',
        press: 70,
        width: 58,
        tempo: 65,
        mentality: 'attacking',
        defensiveLine: 68,
        compactness: 48,
        focus: 'wide',
    },
    awayTactics: {
        formation: '4-3-3',
        style: 'possession',
        mentality: 'balanced',
    },
});

const snapshots = engine.simulate(90 * 60);
const finalSnapshot = snapshots[snapshots.length - 1];
const report = new RealTimeReporter(engine).getReport();

console.log(finalSnapshot.score);
console.log(report.headline);
```

`simulate()` returns every snapshot generated by the match. A full 90 minute match with the default `tickSeconds` of `0.25` creates about 21,600 snapshots, plus extra snapshots for added time. For a text-only game, you usually need the event log and final snapshot, not every snapshot.

## Starters and Substitutes

`RealTimeEngine` uses:

- the first 11 players in `team.players` as starters
- players 12 to 16 as the bench
- generated fallback bench players if fewer than five substitutes are supplied

For a manager game, pass the selected line-up in match order:

```ts
const matchDayPlayers = [
    goalkeeper,
    leftBack,
    leftCenterBack,
    rightCenterBack,
    rightBack,
    leftMidfielder,
    leftCenterMidfielder,
    rightCenterMidfielder,
    rightMidfielder,
    leftForward,
    rightForward,
    substituteKeeper,
    substituteDefender,
    substituteMidfielder,
    substituteWinger,
    substituteForward,
];

const team = new Team(true, 'Northbridge FC', matchDayPlayers);
```

The engine places players from their `Position` values. The `formation` string is stored as tactical metadata and helps describe the plan, while the actual role layout comes from each player's position.

## Tactics

The engine accepts partial tactics. Missing values are filled from the selected style preset and the default balanced setup.

Available styles:

- `balanced`
- `possession`
- `direct`
- `counter`
- `low_block`
- `high_press`

Numeric tactical values are clamped from `0` to `100`.

```ts
const tactics = {
    formation: '4-2-3-1',
    style: 'counter',
    press: 42,
    width: 50,
    tempo: 68,
    mentality: 'defensive',
    defensiveLine: 38,
    compactness: 64,
    focus: 'central',
};
```

## Text Commentary

The real-time engine does not force one commentary style. It returns structured events, and your game can translate those events into the tone you want.

```ts
import type { RealTimeMatchEvent } from '$simulator/RealTimeEngine';

function eventToText(event: RealTimeMatchEvent): string | null {
    const minute = Math.max(1, Math.ceil(event.time / 60));
    const player = event.player?.info.name;
    const team = event.team?.name || event.teamSide;

    if (event.type === 'goal') {
        return `${minute}' Goal for ${team}. ${player} scores.`;
    }

    if (event.type === 'shot') {
        return `${minute}' ${player} gets a shot away.`;
    }

    if (event.type === 'save') {
        return `${minute}' Saved by the goalkeeper.`;
    }

    if (event.type === 'yellow_card') {
        return `${minute}' ${player} is booked.`;
    }

    if (event.type === 'substitution') {
        return `${minute}' ${team} make a substitution.`;
    }

    return null;
}

const feed = engine.events
    .map(eventToText)
    .filter((line): line is string => Boolean(line));
```

For a text based manager game, do not show every pass. Filter for high-signal event types:

- `goal`
- `shot`
- `save`
- `miss`
- `blocked_shot`
- `penalty`
- `red_card`
- `yellow_card`
- `injury`
- `substitution`
- `tactical_change`
- `half_time`
- `full_time`

## Live Match Screen

If you want managers to make changes during the match, drive the engine with `tick()` or simulate in chunks. Do not call `simulate(90 * 60)` to completion before asking for tactical decisions.

```ts
engine.start();

while (engine.state.time < 60 * 60 && engine.state.period !== 'ended') {
    const slice = engine.tick();

    slice.events.forEach((event) => {
        // Push important events to the browser through your websocket.
    });
}

engine.applyTacticalChange('home', {
    style: 'direct',
    tempo: 78,
    mentality: 'attacking',
}, 'manager_chasing_game');

engine.simulate(90 * 60);
```

You can also change one player's role. Use the simulated player id from `engine.state.players` or from the latest snapshot.

```ts
import { Position } from '$simulator/enums/Position';

const player = engine.state.players.find((candidate) => {
    return candidate.side === 'home' && candidate.player.info.number === 8;
});

if (player) {
    engine.applyRoleChange(player.id, Position.COM, 'manager_changed_shape');
}
```

## Browser Replay

Snapshots contain pitch coordinates for all players and the ball. The pitch uses meters:

- length: `105`
- width: `68`
- `x` runs from one goal to the other
- `y` runs touchline to touchline

Convert coordinates to percentages for a browser pitch:

```ts
function pitchX(x: number): number {
    return x / 105 * 100;
}

function pitchY(y: number): number {
    return y / 68 * 100;
}
```

For a text manager game, a full replay is optional. A practical compromise is to store only event times and replay windows around goals:

```ts
const goalReplays = engine.events
    .filter((event) => event.type === 'goal')
    .map((event) => ({
        goalTime: event.time,
        startTime: event.replayWindow?.startTime ?? Math.max(0, event.time - 12),
        endTime: event.replayWindow?.endTime ?? event.time + 4,
    }));
```

## Persisting Events

Project simulator events into plain data before saving.

```ts
import type { RealTimeMatchEvent } from '$simulator/RealTimeEngine';

function toStoredEvent(event: RealTimeMatchEvent) {
    return {
        type: event.type,
        time: event.time,
        minute: Math.max(1, Math.ceil(event.time / 60)),
        teamSide: event.teamSide ?? null,
        teamName: event.team?.name ?? null,
        playerName: event.player?.info.name ?? null,
        playerNumber: event.player?.info.number ?? null,
        secondaryPlayerName: event.secondaryPlayer?.info.name ?? null,
        outcome: event.outcome ?? null,
        score: event.score,
        position: event.position,
        fieldZones: event.fieldZones,
        attackPattern: event.activeAttackPattern,
        chanceQuality: event.chanceQuality ?? null,
        possessionId: event.possession.id,
    };
}
```

Store the match seed with the result. That lets you reproduce the same match if the same squads, tactics, and engine version are used again.

## Server or Browser?

For a manager game with persistent saves, run official match simulations server-side. This prevents tampering and keeps the database update flow simple.

Browser-side simulation is still useful for:

- local demos
- previews
- sandbox tactical testing
- non-authoritative replay rendering

## Post-Match Game Updates

The simulator does not update your save game. After a match, your own game layer should apply:

- fixture result
- league table points and tie-breakers
- player appearance records
- goals, assists, cards, injuries, and suspensions
- fatigue and fitness changes
- morale and form changes
- finances, attendance, fan reaction, and board confidence

Use simulator output as the evidence for those updates.
